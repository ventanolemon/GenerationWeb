# Статистика и аналитика

Схема событий и ролевые срезы. Принцип: на старте — те же реляционные
таблицы основной Postgres + материализованные представления; отдельное
OLAP-хранилище — только когда объём (миллионы событий) это оправдает.
Шов для переезда — события уже нормализованы, экспорт тривиален.

## 1. Источники событий

| Событие | Таблица | Кто пишет |
|---|---|---|
| Попытка/ответ студента | `attempts` (client_uuid, user, partition, assignment?, payload, correct, device) | web сразу; десктоп — через sync-push (офлайн-попытки доезжают с задержкой — это норма модели) |
| Генерация варианта | `generation_events` (user, partition, source: web/desktop, wall_ms) | generator_service |
| Слово-тренажёр | `word_stats` (уже существует в SQLite — переезжает как есть) | движок через WordStatsStore |
| Контур | переходы статусов в `contour_jobs` + `corpus_records` | contour_service |
| Sync | `devices.last_sync_at` + счётчики в лог | web_layer |

`attempts.payload` — JSONB с фидбеком блоков (структурированный diff
WordCorrectionBlock и т.п. уже сериализуется `to_dict()` — переиспользуем
как есть).

## 2. Ролевые срезы (выводятся из RBAC-связей, отдельных прав нет)

- **Студент**: только свои `attempts`/`word_stats` — прогресс по
  предметам, динамика, слабые темы (агрегат по partition/subject).
- **Преподаватель**: агрегаты по СВОИМ группам (`teacher_groups` →
  `group_members` → attempts) в разрезе назначений: кто решал, доля
  верных, распределение попыток; плюс здоровье своих партиций
  (из `generation_events`: частота использования, wall_ms, доля ошибок
  генерации — прямая связь с флагами F1/F3 из таксономии критика).
- **Админ**: всё + метрики контура (джоб за период, доля
  approved/escalated по темам — та самая «частота эскалаций как метрика
  качества заземления» из контракта петли) + рост корпуса по kind/tags.

## 3. Материализованные представления (первый набор)

- `mv_group_progress` — группа × предмет × неделя: попытки, верные, ученики.
- `mv_partition_health` — партиция: генерации, среднее wall_ms, ошибки.
- `mv_contour_funnel` — период: created → approved/rejected/escalated.

Обновление по расписанию (раз в N минут) — учебная аналитика не
realtime; это дешевле триггеров и не трогает горячие таблицы.

## 4. Приватность и объём

- Студент никогда не видит чужих сырых попыток; преподаватель — только
  своих групп; сырые `payload` в UI преподавателя показываются точечно
  (разбор конкретной попытки), агрегаты — по умолчанию.
- `attempts` — самая растущая таблица: партиционирование по месяцу
  заложить в DDL сразу (дешево сейчас, больно потом).
