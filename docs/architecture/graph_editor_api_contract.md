# Граф-контракт: один язык, два канваса

Визуальный редактор графов нужен и в десктопе (есть, QGraphicsScene), и
в вебе (будет). Это НЕ два языка: `core/graph/` (GraphSpec +
GraphExecutor + NodeRegistry) — единственный источник истины, оба
канваса — тонкие view над одним API.

> **Статус реализации (Фаза 2 — серверная сторона API, реализовано).**
> Движок `core/graph/` (117 типов узлов) портирован из десктопа в GenerationWeb
> **байт-в-байт** (25/25 модулей грузятся headless; 11 граф-примеров исполняются
> и сериализуются через web-блоки `to_dict()`). Реализованы §2-эндпоинты
> `GET /graph/catalog`, `POST /graph/validate`, `POST /graph/preview`
> (`generator_service/routers/graph.py` — тонкая обёртка над headless
> `core/graph_api.py`), плюс переиспользуемый `core/graph_probe.py` (probe для
> `/graph/preview` и будущего contour_service). Правка контракта применена:
> upsert партиций расширен до `constracted ≤ 4`. `catalog_version` =
> sha256 отсортированных type_id (первые 16 hex) = `165616006ce6e373` —
> совпадает с provenance корпуса, т.е. набор узлов идентичен десктопному.
> probe в этой фазе покрывает SYM-флаги D2/F1/F2/F3/B4 (детерминированные);
> HYBRID/LLM-коды таксономии — зона агента-критика в contour_service.
> Проверено: 7/7 headless (`core/test_graph_api.py`) + все три эндпоинта через
> FastAPI TestClient. Замечание по окружению:
> полный старт сервиса требует Python 3.12+ из-за pre-existing f-строки в
> `exercises/linal/ex2_d.py` (не в graph-тракте); graph-модули 3.11-совместимы.

> **Статус реализации (Фаза 3 — веб-канвас, скелет реализован).**
> `frontend/src/graph-editor/` — работающий скелет редактора: модель состояния
> (источник правды = сам `GraphSpec.to_dict()`-JSON; тело `repeat`/`map` —
> вложенный GraphSpec в `params.body`, правки адресуются путём — рекурсия
> держится данными, см. `model.ts`), холст (pan/zoom, перенос узлов, провода-
> безье с подсветкой совместимости по `port_types`+`conversions`), палитра из
> `GET /graph/catalog`, инспектор одной формой по `params_schema`, один рабочий
> уровень вложенности (крошки, «↑ Наверх», запрет TASK-узлов в теле), живой
> `/graph/validate` с дебаунсом, предпросмотр `/graph/preview` через
> существующий `<BlockRenderer>`, сохранение существующим upsert партиций
> (`constracted=4`). Динамические порты (formula/text/template/var_dict/
> block_list/repeat/map-туннели) продублированы в TS по правилам §2 — путь,
> который контракт явно разрешает. Проверено в браузере (Chromium):
> physics_force собран руками через UI, добавлен repeat с телом, сохранён,
> страница перезагружена — граф восстановился идентично (включая тело цикла);
> пример из `exercises/graph_examples` открывается и валиден; `/generate` по
> сохранённой партиции отдаёт задание.
>
> Два дополнения за пределами текста контракта, всплывшие на интеграции:
> (1) `bootstrap.build_registry` не имел ветки `constracted == 4` — партиция-
> граф сохранялась, но была «мёртвой» (нет генератора). Портированы
> `exercises/graph/` (`GraphConstructorGenerator`, калька с десктопа) и ветка
> `_register_graph` — это перенос Фазы 1 graph_addon.md, без него «Сохранение»
> из §2 не имеет смысла. (2) dev-режим `VITE_API_DIRECT=1` в vite.config.ts:
> весь `/api` проксируется прямо на FastAPI (без ASP.NET) — для окружений без
> dotnet; продакшен-маршрут (`/api` → web_layer) не тронут, `/api/graph` в
> любом режиме идёт напрямую в generator_service.

## 1. Закрепление формата (правила, не пожелания)

1. Проводная форма графа — ровно `GraphSpec.to_dict()`
   (`core/graph/spec.py`): `{version, nodes[{id,type,params}],
   edges[{from:"n:port", to:"n:port"}], meta}`. Никаких «веб-полей» вне
   `meta`; экранные координаты — `meta.layout`, движок их игнорирует
   (уже так на десктопе).
2. Каталог узлов НИГДЕ не хардкодится клиентом — только из API §2.
   Палитра десктопа уже строится из `NodeRegistry.palette()`; веб
   получает тот же материал по сети.
3. `catalog_version` (хэш выгрузки реестра — тот же, что в
   провенансе корпуса) присутствует в ответах каталога/валидации;
   клиент с устаревшим кешем каталога обязан его перезагрузить.
4. Контрактный тест в CI: граф, сохранённый десктопом, проходит
   `/graph/validate`; граф, собранный вебом, открывается десктопным
   редактором. Симметрия — предмет теста, а не везения.

## 2. API (graph-роутер в `generator_service`)

Роутер живёт в `generator_service` — там уже импортирован `core`, и все
три операции детерминированные (это НЕ контур; LLM здесь нет).

### GET `/graph/catalog`
```json
{
  "catalog_version": "165616006ce6e373",
  "port_types": [{"id": "number", "color_hint": "…"}, …],
  "conversions": [{"from": "expr", "to": "number", "via": "expr_eval"}, …],
  "nodes": [{
      "type_id": "random_natural", "category": "source",
      "display_name": "…", "description": "…",
      "inputs":  [{"name": "…", "type": "number", "required": true}],
      "outputs": [{"name": "out", "type": "number"}],
      "params_schema": { "…": "как PARAMS_SCHEMA узла, дословно" }
  }, …]
}
```
Источники: `NodeRegistry.palette()`, `port_types.py`, `conversions.py`.
Динамические порты (var_dict, formula, туннели repeat) каталогом не
выражаются — клиент запрашивает их пересчёт валидацией (§ ниже) или
реализует те же правила локально по `params_schema` (как десктоп).

### POST `/graph/validate` → `{ok, errors[], result_node, catalog_version}`
Оборачивает сборку `GraphExecutor(spec)`; `errors[]` — ДОСЛОВНЫЕ тексты
`GraphValidationError` (они уже человекочитаемы и называют узлы по id —
тот же принцип, что в repair-петле контура). `result_node` — id
финального TASK-узла (бейдж «ВЫХОД» на любом канвасе).

### POST `/graph/preview` `{graph, seeds:[0,1,2]}` →
```json
{"runs": [{"seed": 0, "statement": [BlockJSON…], "answer": [BlockJSON…],
            "attempts": 1, "wall_ms": 12, "error": null}]}
```
Блоки — через существующий `Block.to_dict()`: превью графа на вебе
БЕСПЛАТНО рендерится уже написанным `<BlockRenderer>`. Это тот же
probe-механизм, что у контура (`closed_loop_contract.md` §3) — одна
инфраструктура, два потребителя; контур зовёт её импортом, веб — по HTTP.
Лимиты: seeds ≤ 8, таймаут на прогон — превью не должно уметь DoS-ить
сервис тяжёлой символикой.

### Сохранение
НОВЫХ эндпоинтов нет: существующий upsert партиций с
`constracted=4`, граф — в `generation_params`. Конкретная правка
текущего кода: `generator_service/routers/partitions.py::
UpsertPartitionRequest.constracted` сегодня ограничен `le=3` — расширить
до 4 (веб сейчас физически не может сохранить граф-партицию).

## 3. Кто чем занимается

| Забота | Где |
|---|---|
| Семантика графа, типы, валидация, исполнение | только сервер (core/graph) |
| Раскладка узлов, провода, выделение | канвас (каждый свой) |
| Палитра, формы параметров | генерируются из каталога (data-driven на обоих) |
| Подсветка совместимости портов | клиент по `port_types`+`conversions` из каталога; спорные случаи добьёт validate |

## 4. Риски веб-порта (названы для оценки объёма, решения не здесь)

1. **Вложенные подграфы** `repeat`/`map`/`case`: десктоп держит стек
   холстов и LabVIEW-рамки; в вебе это самая дорогая часть (кандидат —
   React Flow с subflows, но это выбор реализации, не контракта).
2. **Живой пересчёт динамических портов** (маркеры `#x#` в text/formula,
   туннели) — на десктопе это локальный вызов; вебу нужен дребезг
   validate-запросов или дублирование правил портов в TS.
3. **ParamInspector по `params_schema`**: типы `file`/`hidden`
   (words_file и т.п.) на вебе означают загрузку/выбор серверного
   ресурса — отдельный ресурсный API, в этот контракт не входит.
4. Рендер формул в узлах превью — уже решено (`FormulaBlock.to_dict()`
   отдаёт PNG base64), но объём трафика превью стоит замерить.
