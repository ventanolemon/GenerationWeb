# Принципы расширяемости

Короткий список правил, объясняющих, почему типовая следующая фича —
дешёвая. Каждое правило уже опирается на существующий механизм.

1. **Новый тип задания / предмет** = класс за `GeneratorRegistry` +
   запись в БД. Уже так работает (STANDARD.md десктопа); веб не требует
   правок — блоки полиморфны.

2. **Новый тип блока контента** = класс Block с `to_dict()` (ядро) +
   компонент в `frontend/src/blocks/registry.ts` (веб) — рендереры
   диспатчат по полю `type`, каскадов isinstance нет ни в C#, ни в TS.
   C#-слой не трогается вообще (JsonElement-passthrough).

3. **Новый узел граф-языка** = класс в `core/graph/nodes/` +
   регистрация. Палитры ОБОИХ канвасов подхватят автоматически —
   они data-driven от `/graph/catalog`; меняется `catalog_version`,
   клиенты перезагружают кеш. Ни веб, ни десктоп не правятся.

4. **Новая нечёткая модель** (произношение, генерация текста, другой
   LLM) = провайдер в реестре contour_service + конфиг. Архитектура не
   правится; при появлении тяжёлого локального инференса провайдер
   выносится в отдельный сервис за тем же интерфейсом.

5. **Новая роль/право** = claim в JWT + политика на эндпоинтах
   web_layer. Движки (generator/contour) ролей не знают — новая роль их
   не касается по построению.

6. **Новый клиент** (мобильный, CLI) = потребитель того же web_layer
   API: блоки уже JSON, граф уже JSON, sync-протокол не привязан к
   десктопу (device_id + курсоры).

7. **Смена LLM поколения** в контуре = замена провайдера; контракт
   петли, схема корпуса и eval-набор от модели не зависят — это было
   инвариантом их проектирования.

Анти-правило для ревью: если фича требует править БОЛЬШЕ ОДНОГО из
перечисленных мест — она спроектирована мимо швов; остановиться и
пересмотреть, прежде чем расползтись по слоям.
